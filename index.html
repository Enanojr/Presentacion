Despliegue no-containerizado — Distribution ZIP

ACS 6.2 / 7.x OpenJDK 11 Apache Tomcat 8.5 PostgreSQL 11 ActiveMQ 5.16 Solr 6
Fase 1 · Introducción
Decisión arquitectónica

¿Por qué ZIP vs Docker?

1
Control total: separación explícita de binarios, configuración, datos y logs en rutas independientes del sistema de archivos Linux (FHS).
2
Infraestructura flexible: se despliega sobre bare-metal, VM (VMware / VirtualBox) o Cloud (AWS, Azure) sin dependencia de un runtime de contenedores.
3
Docker: solo para lab/demo: útil para entrenamiento y pruebas rápidas; no recomendado para entornos persistentes de producción.
4
Promotion path: la estructura ZIP facilita promover el mismo despliegue entre entornos — Dev → QA → Prod — con cambios mínimos en setenv.sh.
Fase 1 · Introducción
Visión general

Arquitectura General de Alfresco ACS

Core Platform
Repository + Share
alfresco.war (Repository) y share.war (UI) desplegados sobre Apache Tomcat 8.5 como servidor JEE.
Mensajería
Apache ActiveMQ
Broker de mensajería obligatorio desde ACS v6.0. Gestiona eventos internos entre el repositorio y los microservicios.
Persistencia
PostgreSQL 11
Base de datos relacional para metadata de contenido. ACS crea el esquema automáticamente en el primer arranque.
Servicios satélite
Search · Transform · Sync
Search Services (Solr 6), Transform Service (ATS microservicios) y Sync Service como procesos independientes.

Fase 2 · Prerrequisitos
Infraestructura

Requisitos de Hardware

1
CPU: mínimo 4 cores para entorno de laboratorio; se recomiendan 8+ cores en producción para soportar Tomcat, Solr y ATS simultáneamente.
2
RAM: mínimo 6 GB disponibles (requerimiento del curso); se recomiendan 16 GB+ en producción — Tomcat heap 2 GB, Solr heap 2 GB, OS + ATS el resto.
3
Disco: mínimo 15 GB libres; usar SSD para /var/opt/alfresco (content store) y la BD. El content store crece con los documentos almacenados.
4
Red: puertos abiertos 8080 (ACS/Share), 8161 (ActiveMQ console), 8983 (Solr), 8090/8095/8099 (ATS components). Acceso a internet para descargas.
Fase 2 · Prerrequisitos
Runtime

Java JDK y Apache Tomcat

1
OpenJDK 11: instalación manual desde tar.gz en /opt/jdk-11 — sin usar el package manager para controlar la versión exacta. Definir JAVA_HOME en setenv.sh.
2
Apache Tomcat 8.5.x: único application server certificado desde ACS v6.0. Extraer en /opt/apache-tomcat-{version}.
3
Separación Catalina: CATALINA_HOME = directorio de binarios Tomcat; CATALINA_BASE = /etc/opt/alfresco/tomcat (config + webapps). Esto aísla la configuración de upgrades.
4
Script tomcat.sh: script propio en /opt/alfresco/tomcat.sh que fuente setenv.sh e invoca catalina.sh. Declarado como servicio systemd (tomcat.service).
Fase 2 · Prerrequisitos
Transformaciones

Software de Soporte

1
LibreOffice 6.3.x: convierte documentos Office a PDF vía JodConverter. Instalar con package manager; ruta exportada como LIBREOFFICE_HOME en setenv.sh.
2
ImageMagick 6.9.x / 7.x: conversión y escalado de imágenes. Exportar IMAGEMAGICK_HOME, IMAGEMAGICK_EXE (convert en v6, magick en v7), y rutas de coders/config.
3
Alfresco PDF Renderer 1.1: convierte páginas de PDF a imágenes para preview. Incluido dentro del Distribution ZIP de ACS en la carpeta alfresco-pdf-renderer/.
4
Todas las rutas se centralizan en /etc/opt/alfresco/setenv.sh — el archivo de variables de entorno único de todo el stack, que se carga en cada script de inicio.
Fase 3 · Base de Datos y Mensajería
Persistencia

Configuración de PostgreSQL

1
Instalación: PostgreSQL 11 via repositorio oficial APT/YUM. Habilitar y arrancar con systemctl enable/start postgresql antes de instalar ACS.
2
Usuario y base de datos: sudo -u postgres createuser -P alfresco y createdb alfresco. ACS no requiere creación manual de tablas — las genera en el primer inicio.
3
Autenticación RPM: en distribuciones RPM, cambiar método en pg_hba.conf de peer/ident a trust para habilitar conexión JDBC desde ACS.
4
Driver JDBC: el conector PostgreSQL ya está incluido en el Distribution ZIP de ACS. Configurar db.url, db.username y db.password en alfresco-global.properties.
Fase 3 · Base de Datos y Mensajería
Crítico desde ACS v6.0

Apache ActiveMQ

1
Obligatorio desde ACS v6.0: el broker de mensajería es un prerrequisito estricto. Sin él, el repositorio no arranca correctamente ni los eventos internos se propagan.
2
Versión certificada: Apache ActiveMQ 5.16.x. Instalar desde tar.gz en /opt/apache-activemq-{version}. Configurar ACTIVEMQ_BASE y ACTIVEMQ_DATA en setenv.sh.
3
systemd service: declarar activemq.service en /etc/systemd/system/. El servicio tomcat.service debe declarar Requires=activemq.service para asegurar el orden de inicio.
4
Verificación: consola web en http://localhost:8161 (credenciales: admin/admin). Confirmar que el broker está en estado running antes de arrancar Tomcat.
Fase 4 · Despliegue Core
Distribution ZIP

Instalación del Repository

1
Descarga: obtener alfresco-content-services-distribution-{version}.zip desde el Alfresco Support Portal o Maven Enterprise Repository (credenciales requeridas).
2
Extracción: unzip -d content-services-{version} en /opt/alfresco. El ZIP contiene alfresco.war, share.war, ROOT.war, _vti_bin.war, drivers JDBC y herramienta alfresco-mmt.jar.
3
Licencia: copiar Alfresco.lic en /var/opt/alfresco/content-services/license/. Licencias por MAJOR.MINOR version; deben coincidir con la versión instalada.
4
Módulos AMP: ejecutar apply_amps.sh para instalar alfresco-share-services.amp (integra Share con el repositorio). Script usa alfresco-mmt.jar install y limpia caché de Tomcat.
Fase 4 · Despliegue Core
Configuración central

alfresco-global.properties

1
Ruta: /etc/opt/alfresco/content-services/classpath/alfresco-global.properties. Es el archivo de configuración principal del repositorio.
2
Rutas esenciales: dir.root (content store), dir.keystore (keystores SSL), dir.license.external (licencia ACS). Todas deben apuntar a /var/opt/alfresco/....
3
Conexión BD: db.driver=org.postgresql.Driver, db.url=jdbc:postgresql://localhost/alfresco, db.username=alfresco, db.password=alfresco1.
4
Transformaciones: local.transform.service.enabled=true (ATS) y legacy.transform.service.enabled=false para usar el Transform Service moderno en lugar del LTS embebido.
Fase 4 · Despliegue Core
UI Colaborativa

Despliegue de la Interfaz: share.war

1
share.war: proporciona la interfaz colaborativa clásica de Alfresco Share. Se despliega junto a alfresco.war dentro del mismo Tomcat en CATALINA_BASE/webapps.
2
Módulo requerido: alfresco-share-services.amp debe instalarse en alfresco.war via apply_amps.sh. Sin él, Share no puede comunicarse con el repositorio.
3
Configuración: share-config-custom.xml en classpath de Alfresco. Define endpoints al repositorio y personalizaciones de la UI. Copiado en la carpeta de extensiones web.
4
Digital Workspace: UI moderna alternativa basada en Angular (digital-workspace.war). Disponible en http://localhost:8080/digital-workspace como complemento o reemplazo de Share.
Fase 5 · Búsqueda y Transformación
Full-text indexing

Alfresco Search Services (Solr): Conceptos

1
Basado en Apache Solr 6: indexa metadata de nodos y contenido full-text de documentos. Opcional pero fuertemente recomendado para búsqueda robusta y escalable en producción.
2
Sin Search Services: ACS usa solo la base de datos relacional para búsquedas, lo que limita las capacidades de full-text y penaliza el rendimiento con volúmenes grandes.
3
Cores Solr: core alfresco indexa el workspace activo; core archive indexa la papelera (opcional). Se crean a partir de la plantilla rerank incluida en el ZIP.
4
Verificación: consola Solr en http://localhost:8983/solr. El campo NumDocs debe superar 500 incluso en un repositorio recién creado (nodos del sistema).
Fase 5 · Búsqueda y Transformación
Integración

Conexión de Solr con el Repositorio

1
Activar subsistema: agregar en alfresco-global.properties → index.subsystem.name=solr6. Por defecto el ZIP usa la BD; este cambio redirige las búsquedas a Solr.
2
SSL simplificado (lab): solr.secureComms=none en alfresco-global.properties y alfresco.secureComms=none en solrcore.properties del template rerank.
3
Datos persistentes: configurar data.dir.root=/var/opt/alfresco/search-services en la plantilla de core para separar índices de los binarios de Solr.
4
systemd: alfresco-search.service con PartOf=alfresco-content.service — Solr se detiene cuando ACS se detiene. El arranque se gestiona vía Requires= en alfresco-content.service.

Fase 5 · Búsqueda y Transformación
Instalación ATS

Configuración del Transform Service

1
Extracción: descomprimir alfresco-transform-service-distribution-{version}.zip en /opt/alfresco/transform-service-{version}. Definir ATS_HOME en setenv.sh.
2
3 scripts de inicio: ats-shared-fs.sh (Shared File Store), ats-atr.sh (Transform Router), ats-ate-aio.sh (AIO T-Engine). Cada uno registrado como servicio systemd independiente.
3
Activar en ACS: local.transform.service.enabled=true en alfresco-global.properties. Configurar hosts via CATALINA_OPTS: -Dats-shared-fs.host=localhost -Dats-tengine-aio.host=localhost.
4
Verificación: cada componente expone un endpoint /transform/config con JSON de capacidades de transformación. Errores de conexión al inicio son normales hasta que todos los servicios arrancan.
Fase 6 · Seguridad y Puesta en Marcha
Hardening

Seguridad: Keystores y Usuario de Servicio

1
Usuario no-privilegiado: crear usuario alfresco sin login: sudo useradd -M -N -g users alfresco. Todos los servicios del stack deben ejecutarse como este usuario, nunca como root.
2
Permisos de escritura: el usuario alfresco necesita permisos en los directorios de datos: chown -R alfresco /var/opt/alfresco y /etc/opt/alfresco/search-services/solrhome.
3
Keystore: ACS genera automáticamente el keystore en /var/opt/alfresco/content-services/keystore en el primer arranque. La ruta se define con dir.keystore en alfresco-global.properties.
4
SSL entre ACS y Solr: para producción usar solr.secureComms=https con certificados mutuos. En lab se simplifica a none para reducir complejidad durante el aprendizaje.
Fase 6 · Seguridad y Puesta en Marcha
Performance

Tuning de Tomcat: Memoria y JAVA_OPTS

1
Tomcat heap: en tomcat.sh → CATALINA_OPTS="-Xms2g -Xmx2g". Igualar -Xms y -Xmx para evitar expansiones de heap en caliente. Ajustar según RAM disponible (regla: ≤50% RAM total).
2
ActiveMQ JVM: heap ligero de -Xms128m -Xmx128m — el broker tiene carga baja. Definir en el script de inicio de ActiveMQ o en su propio setenv.sh.
3
Solr JVM: configurar en solr.in.sh → SOLR_OPTS. Agregar -Dsolr.content.dir=/var/opt/alfresco/search-services/contentstore y parámetros de GC apropiados para el volumen de índices.
4
setenv.sh centraliza todas las variables del stack: JAVA_HOME, TOMCAT_HOME, SEARCH_HOME, ATS_HOME, etc. Un único archivo a actualizar durante upgrades futuros.
Fase 6 · Seguridad y Puesta en Marcha
Go-Live

Secuencia de Arranque y Validación

1
Orden de inicio: PostgreSQL → ActiveMQ → Tomcat+ACS → Search Services → Transform Service. Con systemd y dependencias configuradas en los unit files, el orden se gestiona automáticamente.
2
Comando único: sudo systemctl start alfresco-content arranca todo el stack. Verificar con systemctl list-units --type=service que todos los servicios aparecen como loaded active.
3
Análisis de logs: grep "(ERROR|SEVERE|FATAL)" /var/log/alfresco/*.log. Errores de conexión al ATS T-Engine al inicio son esperados (race condition); se resuelven solos cuando todos los servicios están arriba.
4
Endpoints de validación: /alfresco (repo), /share (UI), /alfresco/service/enterprise/admin (Admin Console), :8983/solr (Solr), :8161 (ActiveMQ console).
Fase 6 · Seguridad y Puesta en Marcha
Operaciones

Buenas Prácticas, Logs y Troubleshooting

1
setenv.sh como fuente única de verdad: centralizar todas las variables de entorno facilita upgrades — solo se actualiza la versión en una variable y todos los scripts la recogen automáticamente.
2
Logs críticos a monitorizar: alfresco.log (repositorio), catalina.out (Tomcat), solr.log (Search Services), ats-atr.log (Transform Router), ats-ate-aio.log (T-Engine).
3
Smoke test: subir un documento en Share → verificar preview PDF (valida ATS) → buscar texto con la barra de búsqueda (valida Solr). Si ambas funcionan, el stack está operativo.
4
No mezclar entornos: la instalación ZIP de lab/Docker no debe usarse como base de producción. Producción requiere VM/bare-metal con tuning de memoria, SSL habilitado y backups de /var/opt/alfresco.
